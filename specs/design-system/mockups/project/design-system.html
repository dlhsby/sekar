01

Three-layer model

Tokens are split into three layers. Components consume only Layer 2 semantic tokens — never Layer 1 primitives directly. Layer 3 platform adapters fill in what RN can't do (e.g. CSS variables, focus-visible).

L1 · PRIMITIVES

Raw values

The vocabulary. Hex codes, px values, font names. No meaning, just the palette.

  • color primary, danger, gray-50…900
  • space xs · sm · md · lg · xl · 2xl · 3xl
  • radius sm · base · md · lg · full
  • shadow xs · sm · md · lg · xl (hard-edge)
  • type display · h1·h2·h3 · body · caption · mono
  • motion press · hover · enter · exit
L2 · SEMANTIC

Meaning

Component-shaped roles. btn-primary-bg, input-border-focus, status-active. This is what every button, input, card reads from.

  • btn primary · danger · ghost (bg/fg/border/shadow)
  • input bg · border · border-focus · border-error
  • card · modal bg · border · shadow · scrim
  • badge · status active · idle · outside · missing
  • surface page · card · raised · info · overlay
  • text primary · secondary · tertiary · disabled · inverse
L3 · PLATFORM

Adapters

Where the platforms diverge. Same semantic name, platform-appropriate implementation.

  • touch mobile 48px · web 44px
  • shadow CSS box-shadow · RN composite Views
  • focus :focus-visible ring · 3px primary border on RN
  • motion CSS transitions · RN Animated
  • nav bottom tab bar · sidebar / topbar
  • scroll native swipe · :hover scrollbar
02

Distribution

One JSON, four outputs. Run npm run tokens:build at the repo root and every consumer is updated atomically. CI fails the PR if a generated file is out of sync.

{ } specs/ui-ux/tokens.json The only file you ever edit. Every other file in this section is auto-generated.
Mobile · RN

fe/mobile/src/constants/generated/tokens.ts

  • Plain JS object — nbColors, nbSpacing, nbShadows, nbType
  • nbTokens.ts wrapper adds backward-compat shims
  • Components import named tokens, not hex literals
Web · CSS

fe/web/src/styles/generated/tokens.css

  • CSS custom properties on :root + [data-theme=dark]
  • Auto-injected via the app's root layout
  • Runtime theme switching, no rebuild
Web · Tailwind

fe/web/tailwind.preset.js

  • theme.extend.colors / spacing / borderRadius / boxShadow
  • Use bg-nb-primary, shadow-nb-md in JSX
  • Compiled-out CSS — no runtime cost
Shared · TS

packages/tokens/index.ts

  • Typed export export const tokens = {...}
  • Used by both mobile + web for non-CSS logic (icon colors, charts)
  • Single import for status helpers (getStatusColor)

08

Dark mode hooks

Layer 2 semantic tokens automatically rebind when [data-theme=dark] is set on <html>. RN: a useColorScheme()-driven ThemeProvider swaps nbColors at the React tree root. Dark mode is opt-in for v2.0 — light theme remains default.

Semantic tokenLight valueDark valueWhy it changes
bg.canvas#F5F0EB · warm stone#0F1411 · deep mossInverts surface base; outdoor app stays warm
bg.surface#FFFFFF#1A201DCards lift off canvas, not pure black
black (border / text)#1C1917#F5F0EBOutline color flips with theme
primary#7FBC8C#95D4A0Lifts brightness on dark bg for AA contrast
status.* bgsoft pastel (#DCFCE7 …)muted dark tint (#1F3D2A …)Backgrounds dimmed; foreground stays vivid
shadow.*offset 0 #1C1917offset 0 #F5F0EBHard-edge shadows still readable
gray-50 … 900stone scaleinverted stone scaleText hierarchy preserved

Status colors stay constant in meaning, not in pixel

Status colors (active green, idle amber, danger red, outside purple) keep their foreground hue across themes — semantics over pixels. Only the soft .bg backings darken. This means a returning user sees the same status meaning regardless of theme.
